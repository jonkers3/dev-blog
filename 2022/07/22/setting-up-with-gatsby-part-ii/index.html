Setting Up a Dev Blog with Gatsby in 2022, Part II: Deployment

Today we will deploying the new Gatsby-based dev blog that we just finished setting up.


1. What We Want: Automatic Builds

The main thing we want is some kind of setup where merging changes to our git repository runs our build process and rebuilds our static assets. After a very brief period of reflection we have decided to deploy our blog using Github Pages. Given there are no secrets in our project, the repository can be public.

2. Create a repo on Github

A new, empty repository was created on Github at https://github.com/jonkers3/dev-blog.

3. Setup our git remote

First, we just need to commit our initial changes:

git commit -a -m 'Initial Commit'

Now, add a git remote pointing to our Github repository.

git remote add origin git@github.com:jonkers3/dev-blog.git

4. Setup SSH for Github

Create an SSH key

ssh-keygen -t ed25519 -C 'me@myemail.com'

Add it to your SSH agent:

eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

Copy the public key to your clipboard:

cat ~/.ssh/id_ed25519.pub

Add the SSH key to your Github account Add SSH Key

We should now be able to authenticate with Github via SSH. This can be verified by running the following command:

ssh -T git@github.com

The final step to push our code to Github is this:

git push --set-upstream origin master

For the time being, we can just commit directly to the master branch. Certainly there are many other ways we could be using git, but for the moment, this actually suits our needs quite well.

We commit our changes, call git push, and all of our changes are pushed to Github instantly.

And that is where things are at now: our files are live in our Github repository.

5. Setup Github Pages

Let's give Github actions a try. This appears to be intended to make it very easy for us to deploy our static site to Github Pages, so we'll try to make use of it.

Create the following file:

.github/workflows/gh-pages.js

name: GitHub Pages


on:
  push:
    branches:
      - master
  pull_request:


jobs:
  deploy:
    runs-on: ubuntu-20.04
    permissions:
      contents: write
    concurrency:
      group: ${{ github.workflow }}-${{ github.ref }}
    steps:
      - uses: actions/checkout@v3


      - name: Setup Node
        uses: actions/setup-node@v3
        with:
          node-version: '18'


      - name: Cache dependencies
        uses: actions/cache@v2
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-


      - run: npm ci
      - run: npm run build


      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        if: ${{ github.ref == 'refs/heads/master' }}
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./public

Simply committing this to the repo gets us a ways toward where we want to be.

In addition, goto the repository settings and enable the Github pages branch.

Now, each time we push changes to Github, a job will be triggered, our site's build process will be run, and the resulting artifacts will be published as our updated website. Heading over to our webpage, we can see that it is there, but it doesn't quite look right.

We have been relying a bit on Gatsby's development server up to this point. We need to change a couple of things to make our static site generation work.

6. Tweak path settings

In order for images to work in our static build, we need to add this to our config file:

gatsby-config.ts

+ pathPrefix: 'dev-blog'


  siteMetadata: ...

Given that our Github Pages setup presently has us in a sub-folder called dev-blog, and we don't have anything other than a blog, the extra blog in the URL is redundant. It turns out that the easiest way to get the desired result is basically just to move our blog index and template files out of the blog folder.

src/pages/index.js

mv src/pages/blog/index.js src/pages/index.js
mv src/pages/blog/{mdx.slug}.js src/pages/{mdx.slug}.js

Fix the imports in these two files to reflect this change:

src/pages/{mdx.slug}.js

- import Code from '../../components/Code'
- import Layout from '../../components/Layout'


+ import Code from '../components/Code'
+ import Layout from '../components/Layout'

src/pages/index.js

- import Layout from '../../components/Layout'


+ import Layout from '../components/Layout'

7. Create a gatsby-node.js config file

gatsby-node.js

const path = require(`path`)


exports.createPages = ({ graphql, actions }) => {
  const { createPage } = actions
  return graphql(`
    {
      allMdx(sort: { order: DESC, fields: [frontmatter___date] }, limit: 1000) {
        nodes {
          frontmatter {
            date(formatString: "DD MMMM YYYY")
            title
          }
          slug
        }
      }
    }
  `).then((result) => {
    const posts = result.data.allMdx.nodes


    posts.forEach((node, index) => {
      createPage({
        path: node.slug,
        component: path.resolve(`./src/pages/blog/{mdx.slug}.js`),
        context: {
          slug: node.slug,
          prev: index === 0 ? undefined : posts[index - 1],
          next: index === posts.length - 1 ? undefined : posts[index + 1],
        }
      })
    })
  })
}

This will now happen as a part of our build process.
